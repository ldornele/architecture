# HyperFleet v0.2.0 → v1.0.0 Upgrade Guide

Migration guidance for operators and API consumers. For the authoritative breaking-change checklist with per-item evidence, see [v1.0.0 Breaking-Change Checklist](./v1.0.0/breaking-changes.md).

> **v1.0.0 requires a fresh database.** All existing v0.2.0 deployments are disposable — the upgrade is a **tear-down-and-redeploy**, not an in-place migration.

**See also:** [Release Process](./hyperfleet-release-process.md) · [API Versioning](../../components/api-service/api-versioning.md)

## Upgrade Procedure

All components — API, Sentinel, **and all adapters** — must be upgraded to v1.0.0 and deployed together. A partial upgrade fails: v0.2.0 API rejects PUT on `/statuses`; v1.0.0 API rejects POST. Adapters that remain on v0.2.0 will not report the `Finalized` condition, causing deletion flows to hang.

1. **Prepare v1.0.0 configuration** — apply changes from [§1](#1-configuration).
2. **Export v0.2.0 resource inventory** — record all cluster and node pool IDs, configurations, and adapter registrations via the API before tear-down. v0.2.0 data is not recoverable after this point.
3. **Audit DELETE call sites** — v0.2.0 DELETE returned `501 Not Implemented`; v1.0.0 returns `202 Accepted` with cascade semantics ([§2.4](#24-breaking-delete-returns-202-accepted)). Code that previously ignored 501 errors will now trigger deletions including cascade-delete of child resources.
4. **Tear down v0.2.0** — stop all components. Existing environments are disposable.
5. **Provision fresh database** — do not reuse v0.2.0 database ([§4.1](#41-breaking-fresh-database)).
6. **Confirm authentication is configured** — v1.0.0 defaults to no authentication ([§1.3](#13-breaking-jwt-enabled-default-changed-to-false), [§1.12](#112-breaking-ocm-sdk-removed)). Verify `jwt.enabled: true` is set before deploying.
7. **Deploy all v1.0.0 components** — API first (initializes schema on the fresh DB), then Sentinel, then adapters.
8. **Re-register resources** — create clusters, node pools via the API using inventory from step 2.
9. **Verify** — run [Post-Upgrade Verification](#post-upgrade-verification).

## 1. Configuration

### 1.1 **BREAKING** - Tracing environment variable renamed

The tracing environment variable was renamed. The Helm default remains `false` (disabled) in both versions.

**Action required:** Replace `HYPERFLEET_LOGGING_OTEL_ENABLED` with `HYPERFLEET_TRACING_ENABLED` in any deployment manifests or scripts that set it.

- **Env var renamed:** `HYPERFLEET_LOGGING_OTEL_ENABLED` (v0.2.0) → `HYPERFLEET_TRACING_ENABLED` (v1.0.0). Old var is deprecated and ignored.

```bash
# v0.2.0 (no longer recognized)
HYPERFLEET_LOGGING_OTEL_ENABLED=true

# v1.0.0
HYPERFLEET_TRACING_ENABLED=true   # or =false to disable
```

> **If you don't act:** If you were setting `HYPERFLEET_LOGGING_OTEL_ENABLED`, it is silently ignored in v1.0.0. Tracing reverts to Helm default (`false`).

See [Tracing Standard](../../standards/tracing.md).

---

### 1.2 **BREAKING** - Sentinel `message_decision` CEL configuration

Decision logic moved from hardcoded Go (v0.2.0) to configurable CEL expressions. v1.0.0 ships a default that already references `condition("Reconciled")` — **no update needed** unless you have a custom config.

**Action required (custom config only):** Replace all references to `condition("Ready")` with `condition("Reconciled")` in custom Sentinel and adapter CEL expressions.

**Adapter CEL also affected:** Adapter precondition and capture CEL expressions that reference `condition("Ready")` must also be updated to `condition("Reconciled")`. The same zero-value failure mode applies. Adapter CEL is configured in the adapter's Helm values under `preconditions` and `capture` blocks — apply the same `Ready` → `Reconciled` substitution.

**Custom config only** — update condition references (Sentinel example; same pattern applies to adapter CEL):

```yaml
# v0.2.0 custom config (BROKEN in v1.0.0 — must update)
message_decision:
  params:
    - name: ref_time
      expr: 'condition("Ready").last_updated_time'    # ❌ WRONG: Returns zero-value
    - name: is_reconciled
      expr: 'condition("Ready").status == "True"'     # ❌ WRONG: Returns zero-value
  result: 'is_reconciled || ...'
```

```yaml
# v1.0.0 custom config (REQUIRED update)
message_decision:
  params:
    - name: ref_time
      expr: 'condition("Reconciled").last_updated_time'    # ✓ CORRECT: Returns updated condition
    - name: is_reconciled
      expr: 'condition("Reconciled").status == "True"'     # ✓ CORRECT: Returns updated condition
  result: 'is_reconciled || ...'
```

> **Warning:** `condition("Ready")` returns a zero-value condition in v1.0.0 (`status=""`, `observed_generation=0`). CEL expressions referencing it silently evaluate to false — Sentinel stops publishing events, adapters stop receiving work. No error is logged.

> **Pre-deploy check:** grep all custom Sentinel and adapter config files for `condition("Ready")` and replace with `condition("Reconciled")` before deploying. If Sentinel is not publishing events after deployment, check CEL configuration first — the silent failure is the most likely cause.

See [Sentinel Design — Message Decision](../../components/sentinel/sentinel.md).

### 1.3 **BREAKING** - JWT `enabled` default changed to `false`

**Action required:** Set `config.server.jwt.enabled: true` explicitly in v1.0.0 Helm values.

Helm chart default for `config.server.jwt.enabled` changed from `true` (v0.2.0) to `false` (v1.0.0).

```yaml
# v1.0.0 — explicitly enable JWT if required
config:
  server:
    jwt:
      enabled: true
      identity_claim: email
```

> **If you don't act:** API accepts unauthenticated requests. Combined with OCM SDK removal ([§1.12](#112-breaking-ocm-sdk-removed)), the API may start with no authentication unless JWT is explicitly configured.

---

### 1.4 **BREAKING** - JWT `identity_claim` for audit fields

**Action required:** Confirm your IdP tokens include the configured `identity_claim` (default: `email`). If your tokens use `preferred_username` instead, set it explicitly:

v0.2.0 hardcoded identity to `username`/`preferred_username`. v1.0.0 uses a configurable claim that defaults to `email`.

```yaml
server:
  jwt:
    identity_claim: preferred_username   # default: email
```

> **If you don't act:** Uses `email` claim. A valid JWT is required for **all** requests (GET and mutating) — no token or invalid token → 401. However, if the token is valid but the configured `identity_claim` is absent from the token: **mutating requests are rejected with 401**; **GETs proceed without caller identity** (no `created_by`/`updated_by` attribution, but the request is not rejected). Verify your IdP tokens include the configured claim before deploying.

### 1.7 **BREAKING** - Sentinel Helm: API client config path moved

**Action required:** Move `config.hyperfleetApi.baseUrl` to `config.clients.hyperfleetApi.baseUrl` in Sentinel Helm values.

`config.hyperfleetApi.baseUrl` moved to `config.clients.hyperfleetApi.baseUrl` in Sentinel Helm values.

```yaml
# v0.2.0
config:
  hyperfleetApi:
    baseUrl: http://hyperfleet-api:8000

# v1.0.0
config:
  clients:
    hyperfleetApi:
      baseUrl: http://hyperfleet-api:8000
```

> **If you don't act:** Sentinel fails to start. Template renders empty base URL; validation fails with `clients.hyperfleet_api.base_url required`.

### 1.8 **BREAKING** - Sentinel Helm: `messageDecision` params format changed

**Action required:** Convert `messageDecision.params` from map format to list-of-objects format (see example below).

`messageDecision.params` changed from map format to list format in Sentinel Helm values.

```yaml
# v0.2.0 (map — BROKEN in v1.0.0)
messageDecision:
  params:
    ref_time: 'condition("Reconciled").last_updated_time'
    is_reconciled: 'condition("Reconciled").status == "True"'

# v1.0.0 (list)
messageDecision:
  params:
    - name: ref_time
      expr: 'condition("Reconciled").last_updated_time'
    - name: is_reconciled
      expr: 'condition("Reconciled").status == "True"'
```

> **If you don't act:** Helm template fails to render. Template iterates with `.name` and `.expr` fields; old map format has no such fields.

### 1.9 **BREAKING** - Sentinel config mount paths changed

**Action required:** Update custom scripts, init containers, or sidecar configs to reference `/etc/hyperfleet/` instead of `/etc/sentinel/`.

Sentinel config mount paths changed from `/etc/sentinel/` to `/etc/hyperfleet/`.

- `/etc/sentinel/config.yaml` → `/etc/hyperfleet/config.yaml`
- `/etc/sentinel/broker.yaml` → `/etc/hyperfleet/broker.yaml`

> **If you don't act:** Custom scripts or containers referencing old path fail with config not found.

### 1.10 **BREAKING** - Sentinel `BROKER_TOPIC` env var removed

**Action required:** If overriding the broker topic, set `broker.topic` in Helm values instead of the `BROKER_TOPIC` env var.

`BROKER_TOPIC` env var is no longer injected by the Sentinel deployment template. Use `broker.topic` in Helm values instead.

> **If you don't act:** Topic override silently ignored, falling back to Helm value.

---

### 1.11 **BREAKING** - Adapter Helm: `broker.type` now required

**Action required:** Set `broker.type` to `rabbitmq` or `googlepubsub` explicitly in adapter Helm values. For RabbitMQ, also set `url`, `queue`, `exchange`, and `routingKey`.

v0.2.0 inferred the broker type; v1.0.0 requires it explicitly and fails at render time if missing.

> **If you don't act:** Helm rendering fails at template evaluation.

---

### 1.12 **BREAKING** - OCM SDK removed

**Action required:** Remove OCM auth config (`server.acl`, `server.authz`) and switch to JWT config (`issuer_url`, `audience`, `identity_claim`).

The OCM SDK was removed. Auth is now handled by a standalone JWT handler. Old OCM-specific config fields (`server.acl`, `server.authz`) are silently ignored.

> **If you don't act:** Old auth config has no effect. Combined with [§1.3](#13-breaking-jwt-enabled-default-changed-to-false) (`jwt.enabled` now `false`), the API may start with no authentication.

---

### 1.13 **BREAKING** - Default log format changed to JSON (Sentinel, Adapter)

**Action required:** Update log parsing pipelines if they expect text format from Sentinel or Adapter.

Sentinel and Adapter `DefaultConfig()` changed from `FormatText` to `FormatJSON`. API was already JSON in v0.2.0 — unchanged.

> **If you don't act:** Log parsing breaks for Sentinel and Adapter output.

## 2. API Contract

> API URI path remains `/api/hyperfleet/v1/...`. Component version bump does **not** change API version prefix. See [API Versioning](../../components/api-service/api-versioning.md).

### 2.1 **BREAKING** - Adapter status reporting: POST → PUT

**Action required:** Change adapter status reporting HTTP method from POST to PUT in adapter YAML configs.

Status reporting uses `PUT` with upsert semantics. `POST` no longer accepted. Method is configured in adapter YAML — partner teams must update their configs.

```bash
# v0.2.0
curl -X POST "${API_URL}/api/hyperfleet/v1/clusters/${CLUSTER_ID}/statuses" \
  -H "Content-Type: application/json" -d @status.json

# v1.0.0
curl -X PUT "${API_URL}/api/hyperfleet/v1/clusters/${CLUSTER_ID}/statuses" \
  -H "Content-Type: application/json" -d @status.json
```

> **If you don't act:** 405 Method Not Allowed.

This is a **method-only** change — the request body (`AdapterStatusCreateRequest`) is unchanged. The `Finalized` condition ([§3.2](#32-breaking-finalized-condition-for-deletion-flows)) is a new condition type value, not a schema change. `reason` and `message` on conditions are now optional ([§2.6](#26-condition-reason-and-message-now-optional)).

See [Adapter Status Contract](../../components/adapter/framework/adapter-status-contract.md).

---

### 2.2 New `Resource` response type for new entity types

New entity types (Channel, Version, WifConfig) use a generic `Resource` type with `kind` discriminator and untyped `spec`. Cluster and NodePool responses are unchanged.

**No action required** for existing Cluster/NodePool consumers.

### 2.3 New endpoints

| Endpoint | Description |
|----------|-------------|
| `GET/POST /api/hyperfleet/v1/wifconfigs` | WIF configuration resources (CRUD) |
| `GET/POST /api/hyperfleet/v1/channels` | Release channel resources (CRUD) |
| `GET/POST /api/hyperfleet/v1/channels/{parent_id}/versions` | Versions nested under channels (CRUD) |
| `POST /api/hyperfleet/v1/clusters/{id}/force-delete` | Force-delete stuck cluster |
| `POST /api/hyperfleet/v1/clusters/{id}/nodepools/{np_id}/force-delete` | Force-delete stuck node pool |
| `DELETE /api/hyperfleet/v1/clusters/{id}/nodepools/{np_id}` | Soft-delete node pool (new) |
| `PATCH /api/hyperfleet/v1/clusters/{id}/nodepools/{np_id}` | Patch node pool (new) |

**No action required** — additive endpoints.

> **Force-delete:** Clusters and node pools only. Requires `Finalizing` state (`deleted_time` set). Force-delete is a destructive admin operation — ensure appropriate RBAC permissions restrict access. See [Force Deletion Design](../force-deletion-design.md).

### 2.4 **BREAKING** - Delete returns 202 Accepted

**Action required:** Audit all code paths that call DELETE endpoints — calls that previously expected 501 will now succeed and may cascade-delete child resources.

In v0.2.0, DELETE returned `501 Not Implemented`. Now functional with two-phase delete:

1. **Soft delete:** Sets `deleted_time`, increments `generation`. Returns `202 Accepted`.
2. **Hard delete:** When all required adapters report `Finalized=True`, records are removed.

> **If you don't act:** DELETE now succeeds where it previously returned 501. Code that ignored 501 errors will now trigger real deletions.

> **Data-loss warning:** Deleting a Cluster always cascade-deletes all NodePools. No per-request override. Preserve NodePools by deleting them individually first.

---

### 2.5 **BREAKING** - `kind` field removed from list responses

The `kind` field was removed from `ClusterList`, `NodePoolList`, `ResourceList`, and `AdapterStatusList` response types. Individual resource responses still include `kind`.

**Action required:** Update any client code that expects `kind` in list response envelopes.

```json
// v0.2.0 list envelope (kind present)
{
  "kind": "ClusterList",
  "items": [
    { "id": "abc-123", "kind": "Cluster", "name": "prod-1", ... }
  ],
  "page": 1, "size": 20, "total": 1
}

// v1.0.0 list envelope (kind removed — items still carry kind)
{
  "items": [
    { "id": "abc-123", "kind": "Cluster", "name": "prod-1", ... }
  ],
  "page": 1, "size": 20, "total": 1
}
```

> **If you don't act:** Parse error if client schema requires `kind` on list responses.

### 2.6 Condition `reason` and `message` now optional

`reason` and `message` on adapter conditions are optional. Still recommended for observability.

**No action required.**

## 3. Statuses Contract

### 3.1 **BREAKING** - Aggregated condition renames

**Action required:** Replace all references to `Ready` with `Reconciled` and `Available` with `LastKnownReconciled` in resource-level condition lookups.

| v0.2.0 | v1.0.0 | Meaning |
|--------|--------|---------|
| `Ready` | `Reconciled` | All required adapters report `Available=True` at current generation |
| `Available` | `LastKnownReconciled` | Sticky — stays `True` across generation bumps until new reconciliation completes |

Adapter-level conditions (`Applied`, `Available`, `Health`) are **unchanged**.

> **Name-collision warning:** Adapter-level `Available` still exists as a per-adapter condition. Resource-level `Available` is now `LastKnownReconciled`. When querying conditions, scope lookups to the correct level.

> **If you don't act:** Reading `Ready` / `Available` from resource conditions returns nil/empty.

See [ADR-0007](../../adrs/0007-conditions-based-status-model.md) (Evolution section).

---

### 3.2 **BREAKING** - `Finalized` condition for deletion flows

**Action required:** Adapters must report `Finalized=True` after cleanup during deletion, or resources remain stuck in `Finalizing` state.

New condition type. Include `{ "type": "Finalized", "status": "True", "reason": "CleanupComplete" }` in the adapter status conditions array after cleanup. API gates hard-delete on all required adapters reporting `Finalized=True`.

> **If you don't act:** Resources stuck in `Finalizing` indefinitely. Use `POST /{id}/force-delete` to clear.

---

### 3.3–3.5 Other status changes (no action required)

- **True/False only in steady state.** Resource-level conditions enforce True/False. `Unknown` may appear transiently before the first adapter report.
- **Mapped conditions.** Resource `status.conditions` may now contain dynamically-created conditions from adapter conditions via CEL rules ([Condition Mapping Design](../../components/api-service/condition-mapping-design.md)).
- **Non-reconcilable resources.** WifConfig, Channel, Version have no `RequiredAdapters` — no `Reconciled`/`LastKnownReconciled` conditions. Only Cluster and NodePool receive aggregated conditions.

## 4. Operating Model

### 4.1 **BREAKING** - Fresh database

**Action required:** Deploy with a fresh, empty database. Do not reuse v0.2.0 database. Re-register all resources via API after deployment.

No v0.2.0 data is reused — all tables start empty. Schema changes include new soft-delete columns on `clusters` and `node_pools`, a new `resources` table for WifConfig/Channel/Version, and the aggregated condition rename from `Ready` to `Reconciled` ([§3.1](#31-breaking-aggregated-condition-renames)). Nothing is migrated from v0.2.0.

> **If you don't act:** Undefined behavior against old schema.

---

### 4.2 **BREAKING** - Cascade delete on child resources

New in v1.0.0. In v0.2.0, DELETE returned `501 Not Implemented` — no cascade existed.

Policies hardcoded per entity type, not configurable:

- **NodePool** → `cascade`: deleting Cluster cascade-deletes all NodePools
- **Version** → `restrict`: deleting Channel blocked while Versions exist

```bash
curl -X DELETE "${API_URL}/api/hyperfleet/v1/clusters/${ID}"
# Cascade-deletes all NodePools automatically
# Returns 409 if restrict-policy children exist
```

> **If you don't act:** Deleting Cluster silently cascade-deletes all NodePools. No per-request opt-out.

---

### 4.3 **BREAKING** - Unknown config fields cause startup errors

**Action required:** Remove v0.2.0-only properties from Sentinel configs (e.g., `messaging_system`). Test config against v1.0.0 binary before deploying.

Unexpected properties in config files trigger a startup error. The API already enforced this in v0.2.0 (`UnmarshalExact`). New in v1.0.0: **Sentinel also moved to strict parsing** — v0.2.0 Sentinel used permissive `Unmarshal` which silently accepted unknown fields.

> **If you don't act:** Sentinel refuses to start, naming the unexpected field. API behavior unchanged (was already strict).

---

### 4.4 **BREAKING** - PgBouncer sidecar replaced by generic sidecars

**Action required:** Rewrite PgBouncer config as a generic `sidecars` list entry. See the Helm chart `values.yaml` for the sidecar format.

The dedicated `database.pgbouncer.*` Helm values tree was removed. PgBouncer is now configured as a generic sidecar entry in the `sidecars` list.

> **If you don't act:** Old `database.pgbouncer.*` values silently ignored. Pod starts without DB proxy sidecar.

## Post-Upgrade Verification

- [ ] Request without a valid JWT to `POST /api/hyperfleet/v1/clusters` returns `401` (confirms JWT is enabled)
- [ ] Request without a valid JWT to `GET /api/hyperfleet/v1/clusters` returns `401` (confirms JWT applies to reads too)
- [ ] `GET /api/hyperfleet/v1/channels` and `GET /api/hyperfleet/v1/wifconfigs` return 200
- [ ] `POST /api/hyperfleet/v1/clusters` returns `201 Created`
- [ ] `PUT /api/hyperfleet/v1/clusters/{id}/statuses` returns `201 Created`
- [ ] `GET /api/hyperfleet/v1/clusters/{id}` shows `Reconciled` and `LastKnownReconciled` (not `Ready`/`Available`)
- [ ] Sentinel produces CloudEvents after resource creation (while `Reconciled` is `False`) — if not, check CEL config for stale `condition("Ready")` references ([§1.2](#12-breaking-sentinel-message_decision-cel-configuration))
- [ ] Traces appear in observability backend (if `HYPERFLEET_TRACING_ENABLED=true` is set)
- [ ] Unknown config field causes startup error
